Build a production-quality full-stack web application called "Beneficial Ownership Change Request Workflow System" that satisfies the requirements of a technical assessment for a Full-Stack Developer role.

IMPORTANT:
The goal is not to build many features. The goal is to demonstrate strong software engineering, backend architecture, authorization, testing, workflow enforcement, audit trails, and clean code.

PROJECT CONTEXT

Open Ownership works in beneficial ownership transparency.

The application should simulate a beneficial ownership information update workflow.

ROLES

1. Applicant

- Creates ownership change requests
- Edits requests while in DRAFT status
- Submits requests for review
- Views request status
- Views reviewer comments

2. Reviewer

- Reviews submitted requests
- Approves requests
- Rejects requests
- Returns requests for changes
- Must provide comments when rejecting or requesting changes

APPLICATION FORM

Fields:

- Company Name (required)
- Registration Number (required)
- Beneficial Owner Name (required)
- Ownership Percentage (required, 0-100)
- Change Reason (required)
- Supporting Notes (optional)

WORKFLOW STATE MACHINE

Statuses:

DRAFT
SUBMITTED
UNDER_REVIEW
APPROVED
REJECTED
CHANGES_REQUESTED

Rules:

- Only the owner may edit a DRAFT
- Applicant may submit only DRAFT applications
- Applicant cannot edit once application leaves DRAFT
- Only Reviewer can transition applications after submission
- Reject requires comment
- Request Changes requires comment
- Illegal transitions return structured errors
- Every transition must be audited

Allowed transitions:

DRAFT -> SUBMITTED

SUBMITTED -> UNDER_REVIEW

UNDER_REVIEW -> APPROVED
UNDER_REVIEW -> REJECTED
UNDER_REVIEW -> CHANGES_REQUESTED

CHANGES_REQUESTED -> DRAFT
DRAFT -> SUBMITTED

AUDIT LOG

Every workflow transition creates an audit record containing:

- User ID
- User role
- Previous status
- New status
- Comment
- Timestamp

Display audit history in the application detail page.
